线程池（ThreadPool）是一种管理和复用线程的机制，旨在优化多线程任务的执行效率，减少线程创建和销毁的开销。以下是线程池的核心概念及使用要点：

1. 线程池的优势

  • 资源复用：复用已创建的线程，避免频繁创建和销毁线程的开销。
  • 控制并发：通过参数限制线程数量，防止资源耗尽或过度竞争。
  • 任务管理：提供任务队列、拒绝策略等机制，增强系统的稳定性。

2. 线程池核心参数

  • 核心线程数（corePoolSize）：线程池长期维持的线程数量。
  • 最大线程数（maximumPoolSize）：线程池允许创建的最大线程数。
  • 任务队列（workQueue）：存放待处理任务的阻塞队列（如 LinkedBlockingQueue、ArrayBlockingQueue）。
  • 空闲线程存活时间（keepAliveTime）：非核心线程的空闲存活时间，超时后回收。
  • 拒绝策略（RejectedExecutionHandler）：队列和线程池满时的处理策略，如抛出异常或丢弃任务。

3. 线程池工作流程

  1. 提交任务时，若当前线程数 < 核心线程数，直接创建新线程执行。
  2. 若线程数 ≥ 核心线程数，任务进入队列等待。
  3. 若队列已满且线程数 < 最大线程数，创建临时线程处理任务。
  4. 若队列和线程数均满，触发拒绝策略。

4. 拒绝策略类型

  • AbortPolicy（默认）：抛出 RejectedExecutionException。
  • CallerRunsPolicy：由提交任务的线程直接执行任务。
  • DiscardPolicy：静默丢弃新任务。
  • DiscardOldestPolicy：丢弃队列中最旧的任务，并重试提交。

5. Java中的线程池实现

通过 ThreadPoolExecutor 类手动配置，或使用 Executors 工厂类创建预设线程池：

  • ==newFixedThreadPool：固定线程数，使用无界队列（适合稳定负载）。==
  • ==newCachedThreadPool：弹性线程数，适合短时异步任务（可能创建过多线程）。==
  • ==newSingleThreadExecutor：单线程顺序执行任务。==
  • newScheduledThreadPool：支持定时或周期性任务。

注意：Executors 提供的部分方法（如 newFixedThreadPool）使用无界队列，需警惕内存溢出风险，建议手动配置参数。

6. ThreadPoolExecutor的7个参数

ThreadPoolExecutor 是 Java 中实现线程池的核心类，其构造函数包含 7 个关键参数，这些参数共同决定了线程池的行为和性能。以下是每个参数的详细说明：

1. corePoolSize（核心线程数）

  • 作用：线程池中长期存活的线程数量（即使线程空闲）。
  • 规则：
    • 初始时线程池中没有线程，任务提交后才会创建线程。
    • 线程数达到 corePoolSize 后，新任务会先进入任务队列。
    • 调用 prestartAllCoreThreads() 可提前创建所有核心线程。
  • 建议：根据任务类型（CPU 密集型或 IO 密集型）设置，通常为 CPU 核心数的 1~2 倍。

2. maximumPoolSize（最大线程数）

  • 作用：线程池允许创建的最大线程数（包括核心线程和非核心线程）。
  • 规则：
    • 当任务队列已满且当前线程数 < maximumPoolSize 时，线程池会创建新线程处理任务。
    • 非核心线程空闲超过 keepAliveTime 后会被回收。
  • 建议：结合系统资源（如内存、CPU）和任务负载设置，避免资源耗尽。

3. keepAliveTime（空闲线程存活时间）

  • 作用：非核心线程在空闲状态下的存活时间。
  • 规则：
    • 仅对超出 corePoolSize 的线程生效。
    • 若线程池允许核心线程超时（通过 allowCoreThreadTimeOut(true)），则核心线程也会受此限制。
  • 建议：短任务场景可设置较小值（如 60s），长任务场景可适当增大。

4. unit（时间单位）

  • 作用：keepAliveTime 的时间单位。
  • 可选值：TimeUnit 枚举（如 SECONDS、MILLISECONDS、MINUTES）。
  • 示例：keepAliveTime = 60 + unit = TimeUnit.SECONDS 表示存活时间为 60 秒。

5. workQueue（任务队列）

  • 作用：存放待执行任务的阻塞队列。
  • 常见类型：
    • **LinkedBlockingQueue**（无界队列）：任务无限堆积，可能导致内存溢出。
    • ==**ArrayBlockingQueue**（有界队列）：需指定容量，队列满时触发拒绝策略。==
    • **SynchronousQueue**：不存储任务，直接提交给线程（需 maximumPoolSize 足够大）。
    • **PriorityBlockingQueue**：支持优先级排序。
  • 建议：优先使用有界队列，避免内存溢出。

6. threadFactory（线程工厂）

  • 作用：用于创建新线程的工厂类，可自定义线程名称、优先级、是否为守护线程等。

7. handler（拒绝策略）

  • 作用：当任务队列已满且线程数达到 maximumPoolSize 时，处理新提交任务的策略。
  • 内置策略：
    • ==**AbortPolicy**（默认）：抛出 RejectedExecutionException。==
    • ==**CallerRunsPolicy**：由提交任务的线程直接执行任务。==
    • ==**DiscardPolicy**：静默丢弃新任务。==
    • ==**DiscardOldestPolicy**：丢弃队列中最旧的任务，然后重试提交。==

关键注意事项

  1. 避免无界队列：使用 LinkedBlockingQueue 时需显式设置容量，否则可能导致内存溢出。
  2. 合理设置最大线程数：过大会导致资源竞争，过小可能无法处理突发流量。
  3. 拒绝策略选择：根据业务容忍度选择丢弃任务或回退到调用者线程执行。
  4. 监控线程池状态：通过 getActiveCount()、getQueue().size() 等方法实时监控。
